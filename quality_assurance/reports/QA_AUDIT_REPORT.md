Дата аудита: 20 июня 2026 г.

## Executive Summary

Проверены Flask backend, авторизация, API, парсер, кэш, история обновлений, Excel-экспорт, UI, responsive-верстка, accessibility, безопасность, отказоустойчивость и производительность.

Итог:

- Критических проблем: 2
- High: 8
- Medium: 12
- Low: 7
- Общая оценка качества: **4/10**
- Готовность к ограниченной внутренней демонстрации: **65%**
- Готовность к рабочему релизу: **35%**

Релиз блокируют подделка пользовательской сессии, раскрытые секреты, нерелевантные юридические материалы, возможность формул в Excel и известные уязвимости зависимостей.

## Объем тестирования

- UI: страницы входа, материалы, история, фильтры, поиск, режимы, пагинация, диагностика, загрузка Excel.
- Responsive: 320, 375, 425, 768, 1024, 1440 и 1920 px.
- Формы: пустые строки, пробелы, табуляция, Unicode, emoji, специальные символы, SQL/XSS/HTML payload, длины 100/500/1000/10000.
- API: все маршруты, разрешенные и запрещенные методы, ошибки параметров, авторизация.
- Security: session forgery, open redirect, CSRF, CORS, XSS, formula injection, path traversal, brute force, security headers.
- Performance: FCP/LCP/TTFB, 10/100/1000/10000 запросов.
- Парсер: живой прогон 34 источников без записи кэша и без email.
- Данные: 41 запись кэша и 42 записи живого прогона.

Ограничения:

- Письмо через реальный Gmail не отправлялось, чтобы не создавать внешний побочный эффект.
- Для сценариев 1000 и 10000 пользователей выполнено соответственно 1000 и 10000 запросов с максимальной параллельностью 200.
- Скриншоты были созданы Chrome; геометрия проверена автоматически, ручная пиксельная проверка изображений была ограничена средой.

## Критические проблемы

### QA-001. Пароли пользователей и SMTP app password хранятся открытым текстом

- Категория: Security / Secrets Management
- Критичность: **Critical**
- Расположение: `users.json:3`, `email_config.json:5-6`; оба файла ранее присутствовали в истории Git.
- Шаги: открыть файлы; выполнить `git log --all -- users.json email_config.json`.
- Ожидаемо: в проекте нет паролей, токенов и начальных учетных данных.
- Фактически: доступны пароли четырех аккаунтов и действующий пароль приложения Gmail.
- Причина: секреты сохранены как конфигурация и добавлены в `.gitignore` уже после коммитов.
- Рекомендации: немедленно сменить SMTP app password и все пользовательские пароли; удалить `_initial_passwords`; хранить секреты в переменных окружения/Secret Manager; очистить историю через `git filter-repo`; проверить форки и резервные копии.

### QA-002. Сессию Flask можно подделать без знания пароля

- Категория: Authentication / Session Security
- Критичность: **Critical**
- Расположение: `app.py:48`.
- Шаги: подписать cookie `{"user":"adm24"}` стандартным ключом `change-this-secret-key-before-production`; запросить `/api/status`.
- Ожидаемо: поддельная cookie отклоняется с 401.
- Фактически: API вернул 200 и признал пользователя авторизованным.
- Причина: используется публично известный fallback `SECRET_KEY`.
- Рекомендации: убрать fallback; прекращать запуск без `AI_MONITOR_SECRET_KEY`; создать случайный ключ не менее 32 байт; завершить все существующие сессии.

- Категория: Business Logic / Data Quality
- Критичность: **High**
- Расположение: `app.py:654-689`.
- Шаги: загрузить `cache.json`; повторно вызвать текущий `passes_article_filter` для каждой записи.

### QA-004. Свежий парсер принимает нерелевантные записи как юридические материалы об ИИ

- Категория: NLP / Legal Relevance
- Критичность: **High**
- Шаги: выполнить живой сбор; проверить Federal Register.
- Ожидаемо: строгий режим содержит акты/проекты/юридические действия, явно связанные с ИИ.
- Фактически: в строгий режим попали материалы про медицинские изотопы, патриотическое образование и общие образовательные приоритеты. 18 из 42 результатов пришли из Federal Register.
- Причина: классификация анализирует широкий текст/описание и может связать разрозненные термины, не проверяя предмет основного документа.
- Рекомендации: требовать AI-сигнал в заголовке или основном предмете документа; проверять тип документа и agency; добавить негативные обучающие примеры; использовать отдельный порог для Federal Register; проводить regression-набор экспертно размеченных ссылок.

### QA-005. Stored DOM XSS через URL материала

- Категория: Security / XSS
- Критичность: **High**
- Расположение: `templates/index.html:624`, `scraper.py:306-310`, `scraper.py:405-407`.
- Шаги: проверить URL `https://www.nist.gov/" onmouseover="alert(1)`; валидатор принимает его; `escJs` не экранирует двойную кавычку в HTML-атрибуте `onclick`.
- Ожидаемо: внешние данные не могут создавать HTML-атрибуты/JavaScript.
- Фактически: потенциально создается новый обработчик события.
- Причина: строковая генерация inline JavaScript из внешнего URL.
- Рекомендации: убрать inline `onclick`; использовать `addEventListener` и `dataset`; валидировать URL через строгий allowlist схемы/host/path; запретить управляющие символы и кавычки; включить CSP без `unsafe-inline`.

### QA-006. Excel Formula Injection

- Категория: Security / Export
- Критичность: **High**
- Расположение: `app.py:466-510`.
- Шаги: экспортировать материал с заголовком `=HYPERLINK(...)`; открыть workbook через `openpyxl`.
- Ожидаемо: заголовок остается текстом.
- Фактически: ячейка имеет тип `f` и исполняется как формула при открытии Excel.
- Причина: внешние поля записываются без neutralization.
- Рекомендации: для значений, начинающихся с `=`, `+`, `-`, `@`, добавлять апостроф или задавать текстовый формат; применять ко всем внешним полям и диагностике.

### QA-007. Известные уязвимости закрепленных зависимостей

- Категория: Dependency Security
- Критичность: **High**
- Расположение: `requirements.txt:1-4`.
- Шаги: проверить версии через OSV.
- Фактически: Flask 3.0.3, Flask-CORS 4.0.1 и Requests 2.32.3 имеют опубликованные advisories; Flask-CORS 4.0.1 особенно рискован вместе с глобальным CORS и localhost-приложением.
- Причина: зависимости давно не обновлялись.
- Рекомендации: обновить Flask минимум до 3.1.3, Flask-CORS минимум до 6.0.0, Requests минимум до 2.33.0; добавить Dependabot/Renovate и CI-аудит.

### QA-008. Нет защиты от перебора паролей

- Категория: Authentication
- Критичность: **High**
- Шаги: отправить 30 неверных логинов подряд.
- Ожидаемо: 429, задержка, временная блокировка или CAPTCHA.
- Фактически: все 30 запросов получили 200; ограничения отсутствуют.
- Причина: нет rate limiter и учета неудачных попыток.
- Рекомендации: Flask-Limiter/прокси rate limit; лимит по IP и аккаунту; экспоненциальная задержка; аудит попыток.

### QA-009. Интерфейс сообщает об успехе даже после полного сбоя обновления

- Категория: Reliability / UX
- Критичность: **High**
- Расположение: `app.py:728-787`, `templates/index.html:935-945`.
- Шаги: вызвать исключение внутри `fetch_all_articles`.
- Ожидаемо: статус `failed`, понятная ошибка, старые данные явно помечены.
- Фактически: backend пишет ошибку только в лог, снимает `loading`; frontend показывает «Данные успешно обновлены».
- Причина: API не хранит outcome/error; UI считает окончание процесса успехом.
- Рекомендации: хранить `refresh_status`, `refresh_error`, `finished_at`; возвращать job id; показывать success только при успешном сохранении.

### QA-010. Production запускается встроенным Werkzeug с debug=True

- Категория: Deployment Security / Performance
- Критичность: **High**
- Расположение: `app.py:1076`.
- Шаги: запустить `python app.py`.
- Ожидаемо: production WSGI, debug выключен.
- Фактически: development server и интерактивный debugger.
- Причина: dev-запуск используется как основной.
- Рекомендации: Waitress/Gunicorn; `debug=False`; отдельная production-конфигурация; reverse proxy и HTTPS.

## Medium

### QA-011. Open Redirect после входа

- Категория: Security / Redirect
- Критичность: **Medium**
- Расположение: `app.py:846`.
- Шаги: войти через `/login?next=https://example.org/...`.

### QA-012. CSRF-защита отсутствует, CORS разрешен глобально

- Категория: Security / CSRF / CORS
- Критичность: **Medium**
- Расположение: `app.py:49`, `/api/refresh`, `/logout`.
- Шаги: отправить POST `/api/refresh` с чужим Origin без CSRF-токена.
- Ожидаемо: 403.
- Фактически: сервер принимает запрос и отвечает `started`; CORS отражает Origin.
- Причина: нет CSRF middleware, `CORS(app)` применяется ко всему приложению.
- Рекомендации: убрать CORS либо ограничить конкретными origin/API; Flask-WTF CSRF; проверять Origin; SameSite cookie.

### QA-013. Нет security headers и безопасной конфигурации cookie

- Категория: Security Headers / Clickjacking
- Критичность: **Medium**
- Шаги: открыть `/login`, проверить headers и `Set-Cookie`.
- Фактически: отсутствуют CSP, X-Frame-Options/frame-ancestors, X-Content-Type-Options, Referrer-Policy, Permissions-Policy, HSTS; cookie не содержит `Secure` и явного `SameSite`.
- Рекомендации: Flask-Talisman или middleware; HTTPS; `SESSION_COOKIE_SECURE=True`, `HTTPONLY=True`, `SAMESITE='Lax'/'Strict'`.

### QA-014. Некорректная пагинация вызывает HTTP 500

- Категория: API Validation
- Критичность: **Medium**
- Расположение: `app.py:885-886`.
- Шаги: `/api/articles?page=abc`, `per_page=abc`, `per_page=0`.
- Фактически: 500; отрицательный `per_page` принимается.
- Причина: прямой `int()` и отсутствие нижней границы.
- Рекомендации: schema validation; 400 с описанием; `1 <= per_page <= 50`.

### QA-015. Неизвестный режим возвращает все материалы

- Причина: fallback `split_articles_by_mode` возвращает исходный список.
- Рекомендации: enum validation для mode.

### QA-016. Потеря соединения отображается как «нет материалов»

- Категория: Reliability / UX
- Критичность: **Medium**
- Расположение: `templates/index.html:501-510`.

### QA-017. Гонка записи истории приводит к потере обновлений

- Категория: Data Integrity / Concurrency
- Критичность: **Medium**
- Расположение: `app.py:187-200`, `app.py:588-619`.
- Шаги: два потока одновременно читают пустую историю и сохраняют разные записи.
- Ожидаемо: две записи.
- Фактически: остается одна.
- Причина: read-modify-write без lock и атомарной транзакции.
- Рекомендации: SQLite/PostgreSQL; либо file lock + temp file + atomic replace.

### QA-018. Все аккаунты используют общий кэш, историю и обновление

### QA-019. Еженедельный scheduler ненадежен

## Проверки без найденной уязвимости

- Неавторизованные API возвращают 401; `/` перенаправляет на login.
- SQL/NoSQL injection не применимы: базы данных и пользовательских запросов к БД нет.
- Reflected XSS в login payload не обнаружен.
- Path traversal через history download не воспроизведен.
- Неподдерживаемые HTTP-методы возвращают 405.
- Дубликаты canonical URL в текущем кэше не найдены.
- Обязательные поля у 41 записи заполнены.
- Основные Python-файлы проходят `py_compile`.
