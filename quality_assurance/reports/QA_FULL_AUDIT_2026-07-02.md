## High Bugs

### QA-H-001. Фоновое обновление может упасть после ответа пользователю

- Категория: Functional / Reliability
- Критичность: High
- Шаги воспроизведения: отправить `/api/refresh` с валидным CSRF и завершить процесс/интерпретатор во время фонового сбора.
- Ожидаемый результат: пользователь получает корректный финальный статус, задача не теряется.
- Фактический результат: API сразу возвращает `{"status":"started"}`, а daemon-thread может завершиться ошибкой позже; при тесте получен `RuntimeError: cannot schedule new futures after interpreter shutdown`.
- Предполагаемая причина: сбор выполняется в daemon thread без внешней очереди задач и без lifecycle management.
- Рекомендации: для хостинга вынести обновление в отдельную очередь/worker, либо хранить durable job-status; не использовать daemon для критичных задач.

### QA-H-002. Нет ролей и разграничения прав

- Категория: Authorization
- Критичность: High
- Шаги воспроизведения: войти под любым существующим пользователем и вызвать `/api/refresh`, `/api/history`, `/api/email/status`.
- Ожидаемый результат: опасные операции доступны только администратору/оператору.
- Фактический результат: любой авторизованный пользователь может запускать обновление и смотреть служебный статус email.
- Предполагаемая причина: модель пользователей содержит только логин/пароль без ролей.
- Рекомендации: добавить поле роли в `users.json`, декоратор `admin_required` для refresh/email/status/history download.

### QA-H-003. Production-запуск по умолчанию открывает браузер и включает планировщик

- Категория: Deployment / Operations
- Критичность: High
- Шаги воспроизведения: запустить `python run_server.py` на сервере без переменных окружения.
- Ожидаемый результат: сервер запускается без GUI-действий, фоновые задачи включаются явно.
- Фактический результат: `AI_MONITOR_OPEN_BROWSER` и `AI_MONITOR_ENABLE_SCHEDULER` по умолчанию равны `1`.
- Предполагаемая причина: локальный desktop-сценарий смешан с production-сценарием.
- Рекомендации: для production поставить defaults `0` или документировать обязательные переменные `AI_MONITOR_OPEN_BROWSER=0`, `AI_MONITOR_ENABLE_SCHEDULER=0/1` осознанно.

## Medium Bugs

### QA-M-001. CSP ослаблена через `unsafe-inline`

- Категория: Security / XSS Defense
- Критичность: Medium
- Шаги воспроизведения: открыть ответ `/` и проверить `Content-Security-Policy`.
- Ожидаемый результат: CSP без `unsafe-inline`, скрипты и стили вынесены в статические файлы с nonce/hash.
- Фактический результат: `script-src 'self' 'unsafe-inline'`, `style-src 'self' 'unsafe-inline'`.
- Предполагаемая причина: весь CSS/JS находится inline в `index.html`.
- Рекомендации: вынести JS/CSS в `static/`, убрать inline `onclick`, добавить nonce или hash-based CSP.

### QA-M-002. `target="_blank"` без `rel` в ссылке источника

- Категория: Security / Tabnabbing
- Критичность: Medium
- Шаги воспроизведения: открыть `templates/index.html`, блок `read-more`.
- Ожидаемый результат: внешняя ссылка содержит `rel="noopener noreferrer"`.
- Фактический результат: anchor имеет `target="_blank"` и inline `onclick`, но без `rel`.
- Предполагаемая причина: безопасность добавлена в `window.open`, но не в fallback anchor.
- Рекомендации: добавить `rel="noopener noreferrer"` ко всем внешним ссылкам.

### QA-M-003. Длинный поисковый запрос не ограничен на сервере

- Категория: API / Performance
- Критичность: Medium
- Шаги воспроизведения: запросить `/api/articles?q=` + 10000 символов под авторизованной сессией.
- Ожидаемый результат: 400 или серверное ограничение длины.
- Фактический результат: 200, запрос обрабатывается.
- Предполагаемая причина: `q` не обрезается и не валидируется по длине.
- Рекомендации: ограничить `q` 200-500 символами; возвращать `invalid_query`.

### QA-M-004. Холодный старт API тормозит при параллельных запросах

- Категория: Performance
- Критичность: Medium
- Шаги воспроизведения: выполнить 10 параллельных запросов сразу после импорта/старта приложения.
- Ожидаемый результат: стабильное время ответа.
- Фактический результат: первый smoke дал avg ~5124 ms, p95 ~5754 ms; после прогрева 100/1000 запросов дали p95 ~25/63 ms.
- Предполагаемая причина: конкурентная загрузка/синхронизация `cache.json`.
- Рекомендации: прогружать кэш до открытия порта; не делать ленивую загрузку на первом пользовательском запросе.

### QA-M-005. Email включён, но пароль отсутствует в окружении

- Категория: Functional / Notifications
- Критичность: Medium
- Шаги воспроизведения: проверить `email_config.json` и `.env`.
- Ожидаемый результат: если `enabled=true`, все SMTP-параметры заданы.
- Фактический результат: `enabled=true`, Gmail указан, `.env` пустой, `AI_MONITOR_SMTP_PASSWORD` отсутствует.
- Предполагаемая причина: пароль приложения ещё не создан/не внесён.
- Рекомендации: заполнить `.env` или переменные хостинга; в UI явно показывать `smtp_password_set=false`.

### QA-M-006. Browser responsive audit не автоматизирован в текущем окружении

- Категория: QA Process / UI
- Критичность: Medium
- Шаги воспроизведения: запустить `quality_assurance/artifacts/current_ui_responsive_audit.py`.
