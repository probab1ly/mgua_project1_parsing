# Построчное объяснение работы AI Regulation Monitor

Файл объясняет, как работает проект: где запускается сайт, как происходит авторизация, как парсер собирает данные, как он анализирует релевантность материалов и как результат попадает в интерфейс.

Основные файлы:

- `app.py` — Flask-приложение, авторизация, API, кеш.
- `scraper.py` — источники, парсинг, фильтрация, оценка релевантности.
- `templates/index.html` — интерфейс сайта и JavaScript.
- `templates/login.html` — страница входа.
- `users.json` — разрешенные пользователи.
- `cache.json` — сохраненные результаты парсинга.

---

## 1. Общая логика работы

1. Пользователь открывает сайт.
2. Flask проверяет авторизацию.
3. После входа пользователь видит интерфейс мониторинга.
4. При нажатии «Обновить данные» сайт вызывает `/api/refresh`.
5. `app.py` запускает `fetch_all_articles()` из `scraper.py`.
6. `scraper.py` обходит источники:
   - RSS/Atom-ленты;
   - прямой поиск EUR-Lex;
   - прямой поиск Congress.gov;
   - API Federal Register;
   - HTML-страницы официальных источников.
7. Каждый найденный материал проходит фильтры:
   - URL должен быть валидным;
   - домен должен быть в белом списке;
   - материал должен быть не старше года;
   - в самом заголовке/описании должен явно упоминаться ИИ;
   - должен быть правовой инструмент или правовое событие.
8. Результаты очищаются от дублей.
9. Материалы сортируются по юридической значимости и дате.
10. Результат сохраняется в `cache.json`.
11. Интерфейс получает статьи через `/api/articles`.

## 2. `app.py`: backend и API

### Строки 1-31: импорты

Здесь подключаются стандартные библиотеки Python и Flask.

Особенно важны:

- `json` — чтение и запись `cache.json` и `users.json`.
- `threading` — запуск обновления в фоне, чтобы сайт не зависал.
- `hashlib`, `hmac` — проверка паролей.
- `Flask`, `jsonify`, `render_template`, `request`, `session` — веб-приложение.
- функции из `scraper.py`:
  - `fetch_all_articles()` — главный запуск парсера;
  - `get_stats()` — статистика;
  - `ALL_SOURCES` — полный список источников;
  - `is_ai_relevant()` и связанные функции — повторная проверка кеша.

### Строки 35-47: настройки приложения

`app = Flask(__name__)` создает Flask-приложение.

`app.secret_key` нужен для работы сессий. Без него Flask не сможет безопасно хранить факт входа пользователя.

`CACHE_FILE = Path("cache.json")` — путь к файлу кеша.

`USERS_FILE = Path("users.json")` — путь к файлу пользователей.

`CACHE_VERSION = 8` — версия структуры кеша. Если логика фильтра изменилась, версия повышается, и старый кеш не используется.

`CACHE_LOCK` защищает кеш от одновременной записи из разных потоков.

### Строки 49-55: `_cache`

Это оперативная копия данных в памяти:

- `articles` — список найденных материалов;
- `stats` — статистика по регионам и категориям;
- `last_updated` — время последнего обновления;
- `sources_count` — число источников;
- `loading` — идет ли сейчас сбор данных.

### Строки 60-77: `load_users()`

Функция читает `users.json`.

Логика:

1. Проверяет, есть ли файл `users.json`.
2. Читает JSON.
3. Берет поле `"users"`.
4. Оставляет только записи, где пароль начинается с `pbkdf2_sha256$`.
5. Возвращает словарь `логин -> хеш пароля`.

Регистрации нет. Войти могут только пользователи из этого файла.

### Строки 80-93: `verify_password()`

Проверяет введенный пароль.

Пароль не сравнивается напрямую. Вместо этого:

1. Из сохраненного хеша достаются:
   - алгоритм;
   - число итераций;
   - соль;
   - ожидаемый digest.
2. Введенный пароль снова хешируется через `hashlib.pbkdf2_hmac`.
3. `hmac.compare_digest()` безопасно сравнивает новый digest с сохраненным.

Это защищает от хранения открытых паролей в коде.

### Строки 96-108: `login_required()`

Декоратор защиты страниц и API.

Если пользователь вошел, функция пропускает запрос дальше.

Если пользователь не вошел:

- для `/api/...` возвращается JSON-ошибка `authentication_required`;
- для обычных страниц происходит redirect на `/login`.

### Строки 114-130: `save_cache()`

Сохраняет новые статьи после парсинга.

Что делает:

1. Собирает `payload`.
2. Добавляет `cache_version`.
3. Пересчитывает статистику через `get_stats(articles)`.
4. Записывает время обновления.
5. Обновляет `_cache` в памяти.
6. Записывает JSON на диск в `cache.json`.
7. Снимает флаг `loading`.

### Строки 132-181: `load_cache_from_disk()`

Загружает кеш при старте приложения.

Важные проверки:

1. Если файла нет, возвращает `False`.
2. Если `cache_version` устарел, кеш не используется.
3. Если число источников изменилось, кеш тоже считается устаревшим.
4. Каждая статья заново проходит проверки:
   - домен разрешен;
   - есть явный AI-сигнал;
   - источник Federal Register дополнительно проверяется строгим AI-сигналом;
   - статья релевантна ИИ и праву;
   - дата не старше года.
5. Если у статьи нет `legal_update_type`, он вычисляется заново.
6. Если нет `legal_priority`, он вычисляется заново.
7. Статьи сортируются по:
   - `legal_priority`;
   - дате;
   - `relevance_score`.

Это нужно, чтобы старые или некачественные данные не продолжали отображаться после изменения фильтров.

### Строки 189-200: `refresh_articles()`

Это функция обновления данных.

Что происходит:

1. Ставит `_cache["loading"] = True`.
2. Вызывает `fetch_all_articles()`.
3. Сохраняет результат через `save_cache()`.
4. Если ошибка, пишет ее в лог и снимает `loading`.

### Строки 203-206: `background_refresh()`

Запускает `refresh_articles()` в отдельном потоке.

Это нужно, чтобы кнопка «Обновить данные» не блокировала весь сайт.

### Строки 212-215: `/`

Главная страница.

Доступна только после входа.

Возвращает `templates/index.html`.

### Строки 218-234: `/login`

Страница входа.

При `GET` показывает форму.

При `POST`:

1. Получает логин и пароль.
2. Проверяет логин в `USERS`.
3. Проверяет пароль через `verify_password()`.
4. Если все верно, сохраняет `session["user"]`.
5. Перенаправляет на главную страницу.

### Строки 236-239: `/logout`

Очищает сессию и возвращает пользователя на страницу входа.

### Строки 242-251: `/api/status`

Возвращает состояние приложения:

- идет ли обновление;
- когда были обновлены данные;
- сколько статей;
- сколько источников.

Интерфейс использует это для статуса в шапке и плашки обновления.

### Строки 254-289: `/api/articles`

Главная выдача статей.

Параметры:

- `country` — фильтр по региону;
- `category` — фильтр по категории;
- `q` — поиск;
- `page` — номер страницы;
- `per_page` — сколько материалов на странице.

Фильтрация:

1. Берется копия `_cache["articles"]`.
2. Если выбран регион, остаются только статьи этого региона.
3. Если выбрана категория, остаются только статьи этой категории.
4. Если есть поисковая строка, ищет по `title` и `summary`.
5. Делает пагинацию.

### Строки 292-308: `/api/stats`

Возвращает статистику.

Дополнительно добавляет регионы и категории из `ALL_SOURCES`, даже если по ним пока нет статей. Благодаря этому сайт показывает все регионы и источники, а не только те, что есть в старом кеше.

### Строки 311-325: `/api/sources`

Возвращает полный список источников.

Используется боковой панелью «Источники».

### Строки 328-336: `/api/refresh`

Запускает ручное обновление.

Если обновление уже идет, возвращает `already_loading`.

Если нет:

1. Ставит `loading = True`.
2. Запускает `background_refresh()`.
3. Возвращает `started`.

Автоматического обновления по расписанию сейчас нет.

### Строки 339-345: `/api/filters`

Возвращает все страны и категории.

Важно: берет их не только из статей, но и из `ALL_SOURCES`. Поэтому фильтры показывают весь реестр регионов.

### Строки 353-365: запуск приложения

Если файл запускается как `python app.py`:

1. Проверяет, есть ли пользователи.
2. Пытается загрузить кеш.
3. Если кеша нет, пишет в лог, что автоматический сбор отключен.
4. Открывает браузер.
5. Запускает Flask на `127.0.0.1:5000`.

## 3. `scraper.py`: сбор и анализ данных

### Строки 23-28: основные параметры

- `REQUEST_TIMEOUT = 12` — обычный таймаут HTTP-запроса.
- `SLOW_REQUEST_TIMEOUT = 24` — таймаут для медленных государственных сайтов.
- `MAX_WORKERS = 8` — сколько потоков одновременно собирают источники.
- `RECENT_DAYS = 365` — берем данные минимум за последний год.

### Строки 33-40: `HEADERS`

Заголовки HTTP-запросов.

`User-Agent` имитирует обычный браузер, потому что некоторые сайты блокируют пустые или технические запросы.

### Строки 42-56: `make_session()`

Создает единую `requests.Session`.

В сессию добавляется:

- общий `User-Agent`;
- retry для временных ошибок;
- повтор при кодах `429`, `500`, `502`, `503`, `504`.

### Строка 58: `SESSION`

Одна общая HTTP-сессия для всего парсера.

### Строки 60-68: `fix_url()`

Чистит URL:

- убирает пробелы;
- исправляет лишнюю точку после домена;
- возвращает нормальную ссылку.

### Строки 70-75: `is_valid_url()`

Проверяет, что ссылка имеет:

- схему `http` или `https`;
- домен.

### Строки 80-214: `RSS_SOURCES`

Основной список RSS/Atom-источников.

Туда входят:

- EUR-Lex;
- European Parliament;
- Council of the EU;
- EDPB;
- White House;
- FTC;
- NIST;
- Federal Register;
- OECD;
- UNESCO;
- UK GOV;
- CSET;
- IAPP;
- Japan Digital Agency;
- METI.

Каждый источник описывается одинаково:

- `name` — название;
- `url` — адрес ленты;
- `country` — регион;
- `flag` — флаг;
- `category` — тип источника;
- `description` — пояснение.

### Строки 216-231: `TARGETED_FEED_QUERIES`

Список поисковых запросов для целевого поиска.

Примеры:

- `artificial intelligence regulation`;
- `AI Act`;
- `AI liability directive`;
- `AI regulatory sandbox`;
- `high-risk AI obligations`.

Эти запросы используются, чтобы строить дополнительные поисковые ленты.

### Строки 234-266: `build_targeted_sources()`

Автоматически создает дополнительные источники по запросам.

Например, для каждого запроса создаются:

- Federal Register RSS с ограничением по дате;
- UK GOV Atom search.

Дата отсечения считается так:

```python
datetime.now(timezone.utc) - timedelta(days=RECENT_DAYS)
```

То есть сейчас это последний год.

### Строка 268

```python
RSS_SOURCES.extend(build_targeted_sources())
```

Добавляет автоматически созданные источники к ручному списку.

### Строки 270-305: `ALLOWED_SOURCE_DOMAINS`

Белый список доменов.

Даже если RSS или HTML-страница содержит внешнюю ссылку, она не попадет в результат, если домен не разрешен.

Это защита от случайных ссылок и мусора.

### Строки 307-314: `hostname()` и `is_allowed_source_url()`

`hostname()` достает домен из URL.

`is_allowed_source_url()` проверяет домен по белому списку.

### Строки 316-325: `request_timeout_for()`

Для медленных сайтов дает больше времени.

Например:

- `canada.ca`;
- `industry.gov.au`;
- `senado.leg.br`.

### Строки 327-342: `is_broad_source()`

Определяет широкие источники.

Для таких источников фильтр строже: если сайт широкий, материал должен явно упоминать ИИ в заголовке или описании.

### Строка 344

```python
RSS_SOURCES = [source for source in RSS_SOURCES if is_allowed_source_url(source["url"])]
```

Удаляет из списка RSS все источники, домен которых не разрешен.

## 4. Как парсер анализирует релевантность

Это самая важная часть.

### Строки 371-377: `AI_SUBJECT_TERMS`

Список терминов, которые доказывают связь материала с ИИ.

Примеры:

- `artificial intelligence`;
- `ai`;
- `generative ai`;
- `machine learning`;
- `large language model`;
- `foundation model`;
- `automated decision`;
- `deepfake`;
- `facial recognition`;
- `biometric identification`;
- `high-risk ai`;
- `ai system`.

Важно: слишком широкое слово `algorithm` было убрано, потому что оно давало ложные совпадения.

### Строки 379-388: `LEGAL_RELEVANCE_TERMS`

Общие правовые термины:

- `law`;
- `bill`;
- `regulation`;
- `directive`;
- `standard`;
- `compliance`;
- `guidance`;
- `consultation`;
- `supervision`;
- `certification`.

Эти слова сами по себе недостаточны. Они используются вместе с AI-сигналом и правовым действием.

### Строки 390-398: `STRONG_AI_REGULATION_TERMS`

Сильные фразы, которые почти всегда указывают на регулирование ИИ.

Примеры:

- `ai act`;
- `artificial intelligence act`;
- `ai regulation`;
- `ai liability directive`;
- `executive order on ai`;
- `facial recognition ban`.

### Строки 400-409: `ADOPTION_AND_LEGAL_EVENT_TERMS`

Слова, которые показывают правовое событие:

- `adopted`;
- `approved`;
- `signed`;
- `published`;
- `entered into force`;
- `final rule`;
- `proposed rule`;
- `draft bill`;
- `introduced`;
- `public consultation`;
- `request for comments`;
- `rulemaking`;
- `legislative proposal`.

### Строки 411-418: `HIGH_VALUE_LEGAL_UPDATE_TERMS`

Более ценные правовые события и инструменты.

Они повышают оценку материала.

### Строки 420-426: `LEGAL_INSTRUMENT_TERMS`

Список правовых инструментов:

- акт;
- закон;
- законопроект;
- директива;
- регламент;
- final rule;
- proposed rule;
- notice;
- guidance;
- standard;
- code of practice;
- consultation.

### Строки 428-434: `LEGAL_ACTION_TERMS`

Список действий:

- принят;
- подписан;
- опубликован;
- вступает в силу;
- изменен;
- предложен;
- вынесен на консультацию;
- запрошены комментарии.

### Строки 436-442: lower-case списки

Все списки переводятся в нижний регистр.

Это нужно, чтобы поиск не зависел от регистра.

### Строки 445-448: `has_term()`

Проверяет наличие термина в тексте.

Использует регулярное выражение с границами слова:

```python
(?<![a-z0-9])term(?![a-z0-9])
```

Это важно: `ai` не должен срабатывать внутри другого слова.

### Строки 450-451: `has_any_term()`

Проверяет, есть ли в тексте хотя бы один термин из списка.

### Строки 454-479: `relevance_score()`

Считает числовую оценку релевантности.

Перед обычными баллами функция вызывает `analyze_legal_ai_relevance()`. Это главный rule-based анализатор, который пытается проверить смысловую связку, а не просто отдельные ключевые слова.

Анализатор ищет:

- AI-сигналы в самом материале;
- сильные AI-law формулы;
- правовые инструменты;
- правовые действия;
- близость AI-терминов и правовых терминов в тексте.

В результат анализа входят:

- `is_relevant` — прошел ли материал;
- `confidence` — оценка уверенности;
- `ai_terms` — найденные AI-термины;
- `legal_instruments` — найденные правовые инструменты;
- `legal_actions` — найденные правовые действия;
- `reason` — текстовое объяснение, почему материал принят или отклонен.

Логика:

1. Склеивает `title` и `summary`.
2. Проверяет, есть ли AI-сигнал.
3. Проверяет, есть ли правовой термин.
4. Проверяет, есть ли правовое событие.
5. Проверяет, есть ли правовой инструмент.
6. Проверяет, есть ли действие.
7. Начисляет баллы.

Наибольший бонус дается за связку:

```text
ИИ + правовой инструмент + правовое действие
```

Пример хорошего материала:

```text
AI Act enters into force
```

### Строки 481-490: `is_ai_relevant()`

Главный фильтр релевантности.

Сейчас он опирается на `analyze_legal_ai_relevance()`. Материал проходит, только если анализатор видит осмысленную связку:

```text
есть AI-сигнал
и
есть сильная AI-law фраза
или high-value update
или правовой инструмент + правовое действие
```

Именно здесь решается, попадет ли материал в выдачу.

Важно: поисковый запрос сам по себе больше не доказывает релевантность. Если источник найден по запросу `AI regulation`, но в самом заголовке/описании нет ИИ, материал отбрасывается.

### Строки 492-495: `has_explicit_ai_signal()`

Проверяет, есть ли явное упоминание ИИ в самом материале.

Это защита от ситуации, когда статья попала в поиск по запросу `AI regulation`, но в самой статье ИИ не упоминается.

### Строки 497-507: `has_primary_ai_signal()`

Еще более строгая проверка для широких источников.

Она ищет основные AI-термины:

- `artificial intelligence`;
- `AI`;
- `generative AI`;
- `machine learning`;
- `automated decision`;
- `deepfake`;
- `facial recognition`;
- `AI system`.

Эта функция особенно важна для Federal Register и широких государственных сайтов.

### Строки 509-515: `is_official_ai_law_page()`

Дополнительный fallback для официальных страниц.

Если страница явно про ИИ и правовой инструмент, она может пройти даже если в ней не найдено слово вроде `adopted`.

Это нужно для официальных страниц-обзоров, где формулировки могут быть не новостными.

### Строки 517-537: `legal_priority()`

Присваивает материалу юридический приоритет:

- `4` — принятые акты, вступление в силу, final rules;

- `1` — прочее правовое событие.

Этот приоритет используется в сортировке.

### Строки 540-555: `legal_update_type()`

Присваивает человекочитаемый тип события:

- `Проект / рассмотрение`;
- `Изменение акта`;
- `Стандарт / руководство`;

Этот тип показывается в карточке статьи на сайте.

## 6. Сбор данных из источников

### Строки 683-688: `clean_html()`

Нужно, потому что RSS часто содержит HTML в описании.

### Строки 691-738: `fetch_rss()`

Собирает материалы из RSS/Atom.

Порядок обработки каждой записи:

7. Если источник широкий, требует `has_primary_ai_signal()`.
8. Требует `has_explicit_ai_signal()`.
9. Требует `is_ai_relevant()`.
10. Парсит дату.
11. Проверяет, что дата за последний год.
12. Собирает статью в формат словаря.

5. Проверяет явный AI-сигнал.

## 7. Главный запуск парсера

### Строки 1324-1378: `fetch_all_articles()`

Это центральная функция сбора данных.

Порядок:

1. Создает пустой список `all_articles`.
2. Запускает `ThreadPoolExecutor` на `MAX_WORKERS = 8`.
3. Параллельно запускает:
   - `fetch_rss()` для всех RSS-источников;
   - `scrape_eurlex_ai()`;
   - `scrape_congress_ai()`;
   - `scrape_federal_register_ai()`;
   - `scrape_html_search_source()` для HTML-источников;
   - `scrape_official_page()` для контрольных официальных страниц.
4. Собирает результаты.
5. Удаляет дубли по ссылке.
6. Сортирует по:
   - `legal_priority`;
   - дате;
   - `relevance_score`.
7. Возвращает итоговый список.

Сортировка:

```python
unique.sort(
    key=lambda article: (
        int(article.get("legal_priority", 1)),
        to_datetime(article.get("date", "")),
        int(article.get("relevance_score", 0)),
    ),
    reverse=True,
)
```

То есть сначала идут самые юридически значимые материалы, потом самые свежие.

Считает статистику:

- сколько по странам;
- сколько по категориям;
- топ источников.

## 9. Что именно агрегируется

Проект агрегирует не любые новости об ИИ, а материалы, связанные с правовым регулированием ИИ.

Включаются:

- новые законы;
- законопроекты;
- проекты правовых актов;
- final rules;
- proposed rules;
- notices;
- requests for information;
- public consultations;
- документы о вступлении в силу;
- изменения актов;
- официальные guidance;
- standards;
- codes of practice;
- AI governance frameworks от официальных органов.

Не должны включаться:

- обычные технологические новости;
- бизнес-новости об AI-компаниях;
- статьи про модели без правового контекста;
- материалы, где AI есть только в поисковом запросе, но не в самом тексте;
- документы про алгоритмы без связи с искусственным интеллектом.

## 10. Главная формула анализа

Упрощенно фильтр можно описать так:

```text
Материал проходит, если:

1. В самом title/summary явно есть ИИ.
2. Домен источника разрешен.
3. Материал не старше года.
4. Есть один из вариантов:
   - сильная AI-law фраза;
   - важное правовое обновление;
   - правовой инструмент + правовое действие.
```

Примеры материалов, которые должны проходить:

```text
AI Act enters into force
Bill on artificial intelligence introduced
Request for information regarding artificial intelligence agents
Final rule on automated decision systems
Guidance issued on AI governance
Draft regulation on high-risk AI systems
```

Примеры материалов, которые должны отсеиваться:

```text
Company launches new AI product
AI improves education outcomes
Algorithm used in medical research
Digital transformation strategy without AI-law context
Federal Register notice where AI appears only in unrelated search metadata
```

## 11. Как обновить данные

Автопарсинг по расписанию отключен.

Чтобы обновить данные:

1. Запустить сайт:

```bash
python app.py
```

2. Войти под учетной записью из `users.json`.
3. Нажать «Обновить данные».
4. Дождаться окончания бордовой плашки обновления.

После этого новые статьи будут сохранены в `cache.json`.

## 12. Что делать, если данных мало

Если данных мало, вероятные причины:

1. Слишком строгий фильтр.
2. Источник не ответил по timeout.
3. На странице нет даты, и материал отсекся.
4. RSS-лента изменила формат.
5. В материале нет явного AI-сигнала в title/summary.

Где смотреть:

- логи консоли Flask;
- предупреждения `Источник не ответил`;
- количество статей в `cache.json`;
- `last_updated` в `cache.json`.

---

## 13. Что делать, если данных слишком много или есть мусор

Нужно смотреть, какой фильтр пропустил материал:

- есть ли AI-сигнал;
- есть ли правовой инструмент;
- есть ли правовое действие;
- какой `legal_update_type`;
- какой `legal_priority`;
- какой источник.

Если материал прошел ошибочно, обычно надо поправить:

- `AI_SUBJECT_TERMS`;
- `LEGAL_INSTRUMENT_TERMS`;
- `LEGAL_ACTION_TERMS`;
- `STRONG_AI_REGULATION_TERMS`;
- отдельную логику источника, например Federal Register.
