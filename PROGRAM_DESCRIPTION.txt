ПОДРОБНОЕ ОПИСАНИЕ РАБОТЫ ПРОГРАММЫ
Проект: мониторинг зарубежных и международных правовых новостей об искусственном интеллекте

1. Общее назначение программы

Программа предназначена для закрытого мониторинга зарубежных и международных источников, где могут появляться сведения о новых нормативных правовых актах, законопроектах, регуляторных правилах, официальных руководствах, правовых позициях и иных юридически значимых материалах, связанных с искусственным интеллектом, нейросетями, автоматизированным принятием решений и AI governance.

Главная задача программы - не собирать любые новости про ИИ, а отбирать материалы, где ИИ является предметом правового регулирования. Например:
- принят закон об ИИ;
- опубликован или внесен законопроект об ИИ;
- вступили в силу правила, требования или регламент;
- государственный орган выпустил guidance, framework, policy, code of practice;
- регулятор открыл консультации, изменил правила или опубликовал проект акта;
- официальный орган сообщил о новом правовом режиме для ИИ.

Программа состоит из двух основных частей:
- app.py - Flask-приложение, интерфейс, авторизация, API, кэширование и ручное обновление данных;
- scraper.py - сбор данных из источников, парсинг RSS/API/HTML-страниц, анализ релевантности, сортировка и подготовка карточек новостей.

Также используются стандартные библиотеки Python:

- threading - запуск обновления в фоновом потоке, чтобы интерфейс не блокировался;

- concurrent.futures.ThreadPoolExecutor - параллельный сбор данных из разных источников;
- urllib.parse - кодирование поисковых запросов в URL.

3. Веб-приложение app.py

app.py отвечает за пользовательскую часть программы.

3.1. Авторизация

Сайт закрыт для узкого круга лиц. Регистрация пользователей невозможна.

Пользователи хранятся в файле users.json. В этом файле лежит объект users, где каждому логину соответствует хэш пароля формата:

pbkdf2_sha256$число_итераций$salt$hash

Проверка пароля выполняется функцией verify_password:
- строка хэша разбивается на алгоритм, число итераций, соль и ожидаемый хэш;
- введенный пароль хэшируется через hashlib.pbkdf2_hmac("sha256", ...);
- результат сравнивается с сохраненным хэшем через hmac.compare_digest;
- если логин или пароль неверные, вход не выполняется.

Функция load_users загружает users.json при старте приложения. Если файла нет или он поврежден, авторизация становится невозможной, что защищает сайт от случайного открытого доступа.

Декоратор login_required защищает главную страницу и API. Если пользователь не вошел:
- для обычной страницы он перенаправляется на /login;
- для API возвращается JSON-ошибка authentication_required со статусом 401.

3.2. Основные маршруты Flask

В приложении есть следующие маршруты:

- /
  Главная страница интерфейса. Доступна только после авторизации.

- /login
  Страница входа. При POST проверяет логин и пароль.

- /logout
  Очищает сессию пользователя и возвращает на страницу входа.

- /api/status
  Возвращает состояние системы:
  loading - идет ли обновление;
  last_updated - когда кэш обновлялся последний раз;
  total - сколько материалов сейчас сохранено;
  sources_count - сколько источников подключено.

- /api/articles
  Возвращает карточки материалов. Поддерживает параметры:
  country - фильтр по стране или региону;
  category - фильтр по категории источника;
  q - текстовый поиск по заголовку и описанию;
  page - номер страницы;
  per_page - число материалов на странице, максимум 50.

- /api/stats
  Возвращает статистику по странам, категориям и источникам. Даже если по источнику пока 0 материалов, страна и категория могут отображаться в интерфейсе, потому что они берутся из ALL_SOURCES.

- /api/sources
  Возвращает полный список источников для отображения на сайте.

- /api/refresh
  Запускает ручное обновление данных. Автопарсинг по расписанию убран. Если обновление уже идет, возвращается already_loading.

- /api/filters
  Возвращает список стран и категорий для фильтров. Список строится не только по найденным материалам, но и по подключенным источникам, чтобы на сайте были видны все регионы и направления.

3.3. Кэширование

Данные сохраняются в cache.json. Это нужно, чтобы после перезапуска приложения не терять уже найденные материалы и не запускать парсинг автоматически.

В кэше хранятся:
- cache_version - версия формата и правил фильтрации;
- articles - список найденных материалов;
- stats - статистика;
- last_updated - дата последнего обновления;
- sources_count - количество источников на момент сохранения.

Если изменились правила отбора или список источников, CACHE_VERSION увеличивается. Тогда старый кэш считается устаревшим и не подгружается как актуальный. Это важно, потому что старые данные могли быть собраны по более слабым или слишком строгим правилам.

При загрузке кэша app.py повторно проверяет материалы:
- URL должен относиться к разрешенному домену;
- в заголовке или описании должен быть явный сигнал об ИИ;
- материал должен проходить правовую AI-релевантность;
- дата должна попадать в актуальный период;
- для Federal Register дополнительно требуется первичный AI-сигнал в заголовке или описании.

4. Сбор данных scraper.py

scraper.py отвечает за получение, анализ и сортировку материалов.

4.1. Период сбора

Период актуальности задается константой:

RECENT_DAYS = 365

Это означает, что программа отбирает материалы минимум за последний год. Если дата материала старше 365 дней, он отбрасывается. Если дата не может быть надежно определена, такой материал обычно тоже не попадает в итоговую выдачу, потому что программа должна показывать актуальные новости, а не старые страницы без даты.

4.2. Таймауты и защита от зависания

В scraper.py заданы лимиты:

- REQUEST_TIMEOUT = 12
  Обычный таймаут HTTP-запроса.

- SLOW_REQUEST_TIMEOUT = 24
  Увеличенный таймаут для медленных официальных сайтов.

- HTML_REQUEST_TIMEOUT = 10
  Таймаут для HTML-поиска.

- MAX_WORKERS = 8
  До 8 источников обрабатываются параллельно.

- COLLECT_TIMEOUT = 180
  Общий лимит одного ручного обновления. Если часть сайтов зависла или слишком долго отвечает, сбор не должен вечно блокировать приложение.

HTTP-запросы выполняются через requests.Session. Для сессии настроены:
- User-Agent обычного браузера;
- Accept-Language;
- retry-механизм через HTTPAdapter и urllib3 Retry;
- повтор при кодах 429, 500, 502, 503, 504;
- ограниченное число повторов, чтобы программа не зависала.

5. Как программа получает данные

5.1. RSS/Atom

Функция fetch_rss получает RSS/Atom-ленту через feedparser.

Для каждой записи она берет:
- title - заголовок;
- summary или description - краткое описание;
- link - ссылку;
- дату публикации.

Затем запись проходит проверки:
- источник должен быть в белом списке доменов;
- ссылка должна быть валидной;
- для широких источников требуется сильный AI-сигнал;
- должен быть явный AI-сигнал;
- должна быть правовая релевантность;
- дата должна быть в пределах RECENT_DAYS.

После этого создается карточка article.

5.2. HTML-страницы

Функция scrape_html_search_source получает HTML-страницу через requests, разбирает ее через BeautifulSoup и ищет ссылки.

Для каждой найденной ссылки программа пытается определить:
- заголовок;
- URL;
- контекст вокруг ссылки;
- дату из текста, заголовка или URL.

HTML-источники менее стабильны, чем RSS/API, потому что некоторые сайты:
- формируют результаты через JavaScript;
- блокируют автоматические запросы;
- меняют HTML-разметку;
- не показывают дату в удобном виде;
- долго отвечают или дают таймаут.

Поэтому нулевой результат с HTML-источника не всегда означает отсутствие правовых новостей. Иногда это означает, что сайт технически не отдал данные обычному парсеру.

5.3. Federal Register API

Функция scrape_federal_register_ai использует официальный API Federal Register.

Она отправляет запросы по целевым фразам, например artificial intelligence, AI regulation, automated decision-making и другим связанным правовым формулировкам.

Параметры API включают:
- поисковый термин;
- publication_date >= дата отсечения за последние 365 дней;
- order = newest;
- per_page = 100.

Для каждого документа берутся:
- title;
- abstract;
- document type;
- publication_date;
- html_url.

Federal Register дополнительно фильтруется строже, потому что там много документов, где слова могут встречаться косвенно. Для него требуется, чтобы ИИ был первичной темой, а не случайным словом в тексте.

5.4. EUR-Lex прямой поиск

Функция scrape_eurlex_ai делает поисковые запросы по EUR-Lex:
- artificial intelligence;
- artificial intelligence act;
- AI Act;
- high-risk AI;
- general-purpose AI;
- biometric identification artificial intelligence;
- automated decision-making.

Она парсит результаты поиска и оставляет только материалы, которые проходят общий AI-правовой фильтр и попадают в период актуальности.

5.5. Congress.gov прямой поиск

Функция scrape_congress_ai ищет материалы по законопроектам США, связанным с ИИ. Она нужна, потому что не все материалы Congress.gov удобно получать через обычные RSS-ленты.

6. Как программа понимает, что материал относится к ИИ и праву

В программе используется не нейросетевая модель, а rule-based semantic analyzer - набор правил, словарей терминов и проверок близости слов. Это сделано для прозрачности: можно объяснить, почему материал принят или отклонен.

Главная функция анализа:

analyze_legal_ai_relevance(title, summary="", content="")

Она объединяет заголовок, описание и при необходимости контент в один текст, приводит его к нижнему регистру и ищет несколько групп признаков.

6.1. Группа AI-признаков

Программа ищет термины, указывающие на ИИ:
- artificial intelligence;
- AI;
- generative AI;
- general-purpose AI;
- foundation model;
- machine learning;
- automated decision-making;
- algorithmic decision;
- biometric identification;
- neural network;
- и другие близкие формулировки.

Важно: наличие одного слова AI само по себе не всегда достаточно. Программа дополнительно проверяет юридический контекст.

6.2. Группа правовых инструментов

Программа ищет слова, обозначающие правовой или регуляторный инструмент:
- law;
- act;
- bill;
- regulation;
- rule;
- directive;
- order;
- decree;
- framework;
- guidance;
- policy;
- code of practice;
- standard;
- guidelines;
- proposal;
- consultation;
- enforcement;
- и другие.

6.3. Группа правовых действий и стадий

Программа ищет действия, которые показывают, что речь идет не просто об аналитике, а о юридически значимом событии:
- adopted;
- enacted;
- approved;
- passed;
- entered into force;
- published;
- proposed;
- introduced;
- amended;
- revised;
- updated;
- consultation;
- enforcement;
- issued;
- signed;
- implementation;
- compliance;
- и другие.

6.4. Сильные AI-правовые фразы

Есть отдельная группа сильных фраз. Если они найдены, материал получает высокий уровень доверия:
- AI Act;
- Artificial Intelligence Act;
- AI regulation;
- AI governance framework;
- AI law;
- AI bill;
- high-risk AI;
- general-purpose AI regulation;
- automated decision-making regulation;
- biometric AI rules.

6.5. Проверка близости терминов

Программа не просто смотрит, есть ли слова где-то в тексте. Она проверяет, находятся ли AI-термины рядом с правовыми терминами или действиями.

Для этого используется функция has_nearby_terms. Она ищет позиции терминов в тексте и проверяет, находятся ли они на расстоянии до 280 символов друг от друга.

Это нужно, чтобы не принимать материал, где в одном абзаце говорится об ИИ, а в другом случайно упоминается law или policy по другой теме.

6.6. Расчет confidence

Функция analyze_legal_ai_relevance начисляет условные баллы:
- ИИ в заголовке - сильный плюс;
- ИИ в описании или тексте - плюс;
- сильная AI-правовая фраза - большой плюс;
- найден правовой инструмент - плюс;
- найдено правовое действие или стадия - плюс;
- найден признак законодательной или регуляторной новеллы - плюс;
- ИИ стоит рядом с правовым инструментом - плюс;
- ИИ стоит рядом с правовым действием - плюс;
- официальная правовая страница об ИИ - дополнительный плюс.

Итоговое значение сохраняется в article как analysis_confidence.

6.7. Почему материал принимается

Материал считается релевантным, если:
- есть явный AI-сигнал;
- и одновременно есть хотя бы один из сильных правовых сценариев:
  - найдена сильная AI-правовая фраза;
  - найден признак законодательной или регуляторной новеллы;
  - найден правовой инструмент, правовое действие и близость к ИИ;
  - это официальная правовая страница об ИИ.

То есть программа пытается отбирать не просто новости про технологии, а материалы на пересечении двух тем:

ИИ + право/регулирование/нормативный акт/законопроект/официальный правовой документ.

6.8. Почему материал отклоняется

Материал отклоняется, если:
- в нем нет явного упоминания ИИ;
- есть ИИ, но нет правового инструмента;
- есть ИИ и правовые слова, но нет события принятия, изменения, публикации, рассмотрения или официальной правовой страницы;
- связь между ИИ и правовым событием слишком слабая;
- дата старше 365 дней;
- дата неизвестна и нельзя подтвердить актуальность;
- URL не относится к разрешенному домену.

7. Структура одной найденной карточки

Каждая найденная запись превращается в словарь article.

В нем хранятся:
- id - технический идентификатор на основе ссылки;
- title - заголовок материала;
- summary - краткое описание;
- link - URL материала;
- date - дата публикации или дата документа;
- collected_at - когда программа собрала запись;
- source - название источника;
- country - страна или регион;
- flag - флаг для интерфейса;
- category - категория источника;
- source_description - описание источника;
- relevance_score - оценка релевантности;
- legal_update_type - тип правового события;
- legal_priority - приоритет правовой значимости;
- analysis_confidence - уверенность фильтра;
- analysis_reason - объяснение, почему материал прошел фильтр;
- matched_ai_terms - найденные AI-термины;
- matched_legal_instruments - найденные правовые инструменты;
- matched_legal_actions - найденные правовые действия.

8. Типы правовых событий

Функция legal_update_type определяет тип материала по словам в заголовке и описании.

Примерные типы:
- вступление в силу;
- принятие или утверждение;
- новый законопроект;
- поправки или обновление;
- консультации или обсуждение;
- guidance/framework;
- enforcement или контроль;
- общий правовой материал.

Это помогает пользователю быстрее понять, что именно произошло.

9. Приоритет и сортировка

Функция legal_priority оценивает юридическую значимость материала.

Выше получают:
- законы;
- акты;
- регламенты;
- законопроекты;
- вступление в силу;
- принятие;
- официальные правила;
- regulatory framework;
- enforcement.

После сбора все материалы:
- объединяются;
- дубли удаляются по ссылке;
- сортируются.

Сортировка идет по:
- legal_priority;
- дате;
- relevance_score.

Поэтому в интерфейсе сначала должны идти более новые и более юридически значимые материалы.

10. Защита от случайного мусора

В программе есть несколько уровней защиты:

1. Белый список источников и доменов.
   Программа не берет ссылки с произвольных сайтов, если они не относятся к разрешенным источникам.

2. Проверка URL.
   Невалидные или пустые ссылки отбрасываются.

3. Явный AI-сигнал.
   Материал должен явно упоминать ИИ, artificial intelligence, AI Act, machine learning, automated decision-making или близкие термины.

4. Правовой контекст.
   Материал должен содержать правовой инструмент или правовое действие.

5. Проверка близости.
   AI-термины должны быть связаны с правовыми терминами в тексте, а не встречаться случайно.

6. Дата.
   Материал должен попадать в период последних 365 дней.

7. Повторная фильтрация кэша.
   Даже сохраненные материалы повторно проверяются при загрузке cache.json.

11. Почему иногда источники дают 0 материалов

Это может происходить по нескольким причинам:

- на источнике действительно не было новых правовых материалов об ИИ за последний год;
- сайт не имеет RSS и отдает результаты поиска через JavaScript;
- сайт изменил HTML-разметку;
- сайт ограничивает автоматические запросы;
- сайт отвечает слишком долго и попадает под таймаут;
- на сайте есть материалы про ИИ, но без правового события;
- дата материала не распознана;
- материал старше 365 дней;
- фильтр посчитал связь ИИ и права недостаточно явной.

Особенно часто это относится к HTML_SEARCH_SOURCES. RSS и официальные API обычно надежнее.

12. Ограничения текущего подхода

Текущий анализ является прозрачным и объяснимым, но он не является полноценной языковой нейросетевой моделью. Он использует словари терминов, регулярные выражения, анализ близости слов и систему баллов.

Плюсы такого подхода:
- можно объяснить, почему запись принята;
- легко настроить словари;
- не нужны внешние AI API;
- не передаются данные третьим лицам;
- система работает локально и предсказуемо.

Минусы:
- сложные тексты могут быть отклонены, если юридический смысл выражен нестандартно;
- материалы на языках, отличных от английского, могут хуже распознаваться;
- HTML-сайты могут технически не отдавать данные;
- без полноценной NLP-модели программа не понимает смысл так же глубоко, как эксперт-юрист;
- слишком строгие правила уменьшают количество результатов, а слишком мягкие увеличивают риск мусора.

13. Что именно агрегирует программа

Программа агрегирует не только законы в узком смысле.

Она собирает более широкий круг юридически значимых материалов об ИИ:
- законы;
- регламенты;
- директивы;
- законопроекты;
- проекты правил;
- официальные консультации;
- guidance;
- regulatory framework;
- code of practice;
- policy papers официальных органов;
- документы регуляторов;
- официальные новости о принятии, изменении, вступлении в силу или рассмотрении актов;
- международные правовые и политико-правовые документы по AI governance.

То есть предмет мониторинга - законодательные и регуляторные новеллы в сфере ИИ, а также официальные правовые материалы, которые описывают принятие, изменение, внедрение или обсуждение правового режима ИИ.

14. Как пользователь работает с сайтом

1. Пользователь открывает сайт.
2. Вводит логин и пароль из users.json или переданного файла с учетными данными.
3. После входа видит интерфейс мониторинга.
4. Нажимает кнопку обновления данных.
5. Flask запускает фоновый поток.
6. scraper.py параллельно обходит источники.
7. Каждая запись проходит фильтры.
8. Итоговые материалы сохраняются в cache.json.
9. Интерфейс через API получает статус, статистику и список материалов.
10. Пользователь фильтрует материалы по стране, категории или поисковой строке.

15. Почему обновление сделано ручным

Автоматический парсинг через заданное время убран намеренно. Это сделано потому, что:
- официальные сайты могут отвечать медленно;
- частые запросы могут привести к блокировкам;
- пользователь должен сам контролировать момент обновления;
- при закрытом использовании в вузе удобнее обновлять данные по необходимости.

16. Какие файлы важны

- app.py
  Flask-приложение, авторизация, API, кэш, ручное обновление.

- scraper.py
  Источники, сбор данных, фильтрация, анализ релевантности, сортировка.

- templates/index.html
  Основной интерфейс мониторинга.

- templates/login.html
  Страница входа.

- users.json
  Разрешенные пользователи и хэши паролей.

- cache.json
  Последние собранные и отфильтрованные материалы.

- requirements.txt
  Список Python-зависимостей.

- REPORT.md
  Краткий отчет о проекте, изменениях и логике.

- CODE_EXPLANATION.md
  Более техническое объяснение кода.

17. Итоговая логика программы коротко

Программа берет данные из официальных и профильных зарубежных источников, приводит их к единому виду, проверяет наличие темы ИИ, проверяет наличие правового акта или регуляторного события, отбрасывает старые и случайные материалы, сортирует оставшееся по актуальности и юридической значимости, сохраняет результат в кэш и показывает его авторизованному пользователю через Flask-интерфейс.
