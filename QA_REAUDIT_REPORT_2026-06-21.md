# Повторный полный QA-аудит AI Regulation Monitor

Дата аудита: 21 июня 2026 г.

## Executive Summary

Проверены Flask backend, авторизация, API, парсер, NLP-классификация,
кэш и история обновлений, Excel-экспорт, email-уведомления, UI,
responsive-верстка, accessibility, security, SEO и производительность.

Текущее состояние значительно лучше результата аудита от 20 июня:

- 16 автоматических тестов прошли;
- XSS, SQL payload и HTML injection не исполняются;
- CSRF, open redirect, session forgery и Formula Injection закрыты;
- неавторизованный API возвращает 401;
- некорректные методы возвращают 405;
- `pip-audit` не обнаружил известных уязвимостей зависимостей;
- на ширинах 320–1920 px нет горизонтального переполнения;
- 10 000 read-only запросов завершились без ошибок;
- текущий кэш содержит 31 материал за период 23.05.2025–18.06.2026.

Остаточные проблемы:

- Critical: 1
- High: 3
- Medium: 8
- Low: 5

Общая оценка качества: **7/10**.

Готовность к ограниченной внутренней эксплуатации: **82%**.

Готовность к рабочему релизу с обязательными email-уведомлениями: **68%**.

Релиз блокируют секреты в истории Git и отсутствие достоверного контроля
доставки email. Перед использованием результатов как юридически точной
выборки также требуется исправить несколько ложных строгих классификаций.

## Объем тестирования

- UI: вход, материалы, два режима, поиск, фильтры, пагинация, история,
  диагностика, Excel-ссылки, уведомления и состояние обновления.
- Responsive: 320, 375, 425, 768, 1024, 1440 и 1920 px.
- Формы: пустые значения, пробелы, Unicode, emoji, специальные символы,
  SQL/XSS/HTML payload и длины 100/500/1000/10000.
- API: GET/POST/PUT/PATCH/DELETE для всех маршрутов.
- Security: CSRF, XSS, SQL injection, path traversal, open redirect,
  clickjacking, CORS, cookie, session forgery, Formula Injection.
- Performance: 10, 100, 1000 и 10000 запросов.
- Данные: 31 текущий материал и диагностика 34 источников.

Ограничения:

- реальное письмо не отправлялось, поскольку в процессе отсутствует
  `AI_MONITOR_SMTP_PASSWORD`;
- 1000 и 10000 пользователей смоделированы соответствующим количеством
  запросов с максимальной параллельностью 200;
- регистрация, профиль, комментарии, лайки, подписки, платежи, загрузка
  файлов и роли отсутствуют по проектному замыслу;
- визуальная геометрия использует последний браузерный прогон от 20 июня,
  дополненный статической проверкой текущих шаблонов.

## Critical

### RQA-001. Пароли пользователей и SMTP-пароль остаются в истории Git

- Категория: Security / Secrets Management
- Критичность: **Critical**
- Расположение: коммиты `318fff82eed0`, `7cf91963a931`,
  `e244fb17395f`.
- Шаги воспроизведения:
  1. Выполнить `git log --all -- users.json email_config.json`.
  2. Прочитать старые версии файлов через `git show`.
- Ожидаемый результат: Git не содержит открытых паролей.
- Фактический результат: старая версия `users.json` содержит
  `_initial_passwords`, две версии `email_config.json` содержат
  `smtp_password`.
- Предполагаемая причина: файлы добавили в `.gitignore` уже после коммитов.
- Рекомендации:
  1. Отозвать старый Gmail app password и сменить все исторические пароли.
  2. Очистить историю через `git filter-repo`.
  3. Выполнить force-push только после согласования с участниками.
  4. Проверить форки, архивы и резервные копии.

## High

### RQA-002. Сбой email скрывается, а текущая SMTP-конфигурация не готова

- Категория: Functional / Email / Reliability
- Критичность: **High**
- Расположение: `app.py:454-493`, `app.py:552-579`,
  `app.py:921-923`.
- Шаги воспроизведения:
  1. Оставить `AI_MONITOR_SMTP_PASSWORD` незаданным или вызвать SMTP timeout.
  2. Запустить обновление.
  3. Проверить `/api/status`.
- Ожидаемый результат: пользователь видит, что данные обновлены, но письмо
  не доставлено; ошибка и время попытки сохраняются.
- Фактический результат: `send_email_notification()` возвращает `False`,
  но результат игнорируется; `refresh_status` становится `success`.
  Текущая конфигурация: `enabled=true`, получатель есть, но
  `smtp_password_set=false`, `ready=false`.
- Предполагаемая причина: email считается необязательным побочным действием,
  его результат не включен в модель состояния обновления.
- Рекомендации: хранить `email_status`, `email_error`, `email_sent_at`;
  показывать их в истории и UI; вынести отправку в очередь с повторами;
  уменьшить SMTP timeout с 90 секунд; добавить кнопку тестового письма.

### RQA-003. Ошибки сохранения кэша и Excel не отменяют успешный статус

- Категория: Data Integrity / Reliability
- Критичность: **High**
- Расположение: `app.py:742-795`, `app.py:915-923`.
- Шаги воспроизведения:
  1. Запретить запись в `cache.json` или `update_history`.
  2. Запустить обновление.
- Ожидаемый результат: обновление имеет статус `failed` либо
  `partial_success`, пользователь получает точную причину.
- Фактический результат: `save_cache()` и `save_update_history()` ловят
  исключения и не передают их выше; обновление помечается успешным.
- Предполагаемая причина: функции сохранения логируют ошибку, но не возвращают
  обязательный результат операции.
- Рекомендации: использовать исключения или `Result`; считать обновление
  успешным только после атомарного сохранения кэша и отчета; добавить тест
  read-only диска и нехватки места.

### RQA-004. Часть материалов ошибочно попадает в строгий режим

- Категория: Business Logic / NLP / Legal Relevance
- Критичность: **High**
- Шаги воспроизведения:
  1. Открыть строгий режим.
  2. Найти `NIST Artificial Intelligence Consortium`,
     `AI Pact marks one year of progress...`,
     `EU GPAI Safety Tender`.
- Ожидаемый результат: строгий режим содержит принятые/измененные акты,
  опубликованные проекты и формальные правовые процедуры.
- Фактический результат: консорциум, отчет о прогрессе и государственная
  закупка классифицированы как строгие правовые события.
- Предполагаемая причина: NLP-эвристика связывает слова `AI`, `Act`,
  `guidelines`, `office`, `standards` из широкого текста без надежной
  проверки предмета документа и стадии правового процесса.
- Рекомендации: отдельный negative-класс для tender/consortium/progress;
  требовать legal action в заголовке или первом смысловом абзаце; вести
  экспертно размеченный regression-набор; показывать спорные записи на
  ручную модерацию.

## Medium

### RQA-005. Недельный планировщик сбрасывается при каждом перезапуске

- Категория: Scheduler / Functional
- Критичность: **Medium**
- Расположение: `app.py:956-971`, `run_server.py:12-13`.
- Шаги: перезапускать приложение чаще одного раза в семь дней.
- Ожидаемый результат: проверка выполняется по календарному расписанию.
- Фактический результат: поток каждый раз начинает новый `sleep(7 days)`;
  при регулярных перезапусках автообновление может никогда не произойти.
- Причина: расписание не хранится и не рассчитывается по `last_auto_refresh`.
- Рекомендации: APScheduler/Task Scheduler/Cron с persistent job store;
  при старте проверять просроченный запуск; фиксировать следующий запуск.

### RQA-006. Rate limit хранится только в памяти и применяется только по IP

- Категория: Authentication / Availability
- Критичность: **Medium**
- Расположение: `app.py:97-99`, `app.py:153-173`.
- Шаги: перезапустить процесс после блокировки либо атаковать разные процессы;
  проверить нескольких пользователей за одним NAT.
- Ожидаемый результат: устойчивый лимит по IP и аккаунту с `Retry-After`.
- Фактический результат: перезапуск сбрасывает счетчик; один пользователь
  способен заблокировать вход всем коллегам с общего IP.
- Причина: локальный `defaultdict`, нет внешнего хранилища и account key.
- Рекомендации: Flask-Limiter + Redis; лимиты по IP и логину; журнал попыток;
  `Retry-After`; экспоненциальная задержка.

### RQA-007. Авторизованные ответы разрешено кэшировать браузеру/прокси

- Категория: Security / Privacy
- Критичность: **Medium**
- Шаги: открыть `/` или `/api/status`, проверить `Cache-Control`.
- Ожидаемый результат: `Cache-Control: no-store, private`.
- Фактический результат: заголовок отсутствует.
- Причина: общий middleware добавляет security headers, но не cache policy.
- Рекомендации: для HTML/API/Excel с внутренними данными установить
  `no-store, private`; для статики оставить контролируемый public cache.

### RQA-008. CSP допускает inline JavaScript и inline handlers

- Категория: Security / XSS Defense in Depth
- Критичность: **Medium**
- Расположение: `app.py:129-130`, `templates/index.html:364-986`.
- Шаги: проверить CSP и шаблон.
- Ожидаемый результат: nonce/hash CSP, отсутствуют inline `onclick`.
- Фактический результат: `script-src 'unsafe-inline'`; интерфейс использует
  inline handlers и большой встроенный script.
- Причина: JavaScript не вынесен в отдельный файл.
- Рекомендации: перенести JS в `static/app.js`, заменить handlers на
  `addEventListener`, убрать `unsafe-inline`.

### RQA-009. Нет срока жизни и принудительного отзыва пользовательских сессий

- Категория: Authentication / Session Management
- Критичность: **Medium**
- Шаги: сохранить cookie и использовать ее позднее.
- Ожидаемый результат: ограниченный idle/absolute lifetime и отзыв сессий.
- Фактический результат: `PERMANENT_SESSION_LIFETIME` и server-side revocation
  отсутствуют; смена пароля не завершает старые cookie.
- Причина: stateless Flask cookie sessions.
- Рекомендации: 8–12 часов, server-side session store, `session_version`
  пользователя, отзыв при смене пароля.

### RQA-010. Обновление длится до нескольких минут без детального прогресса

- Категория: UX / Performance / Reliability
- Критичность: **Medium**
- Шаги: нажать «Обновить данные» при медленных источниках.
- Ожидаемый результат: видны количество завершенных источников, текущая стадия,
  прошедшее время и возможность безопасно покинуть страницу.
- Фактический результат: общий spinner; диагностика обновляется после окончания.
  Контрольные сборы занимали 72–100 секунд, глобальный timeout — 180 секунд.
- Причина: состояние `SOURCE_DIAGNOSTICS` не транслируется в job progress API.
- Рекомендации: job id, `/api/jobs/<id>`, `processed/total`, SSE или polling,
  статус email и сохранения отдельными этапами.

### RQA-011. 19 из 34 источников не дали ни одного материала

- Категория: Data Completeness / Integrations
- Критичность: **Medium**
- Шаги: проверить последнюю диагностику.
- Ожидаемый результат: понятно, где действительно нет релевантных актов, а где
  сломалась интеграция.
- Фактический результат: материалы дали 15 источников; статусы:
  23 `ok`, 4 `error`, 5 `timeout`, 2 `unavailable`. У части `ok` результат 0.
- Причина: 403, JS-сайты, нестабильные RSS/HTML и строгая фильтрация.
- Рекомендации: разделить `parsed_zero` и `filter_zero`; хранить raw sample;
  добавить официальные API; мониторить изменение DOM; показывать дату
  последнего успешного результата по источнику.

### RQA-012. Клавиатурный focus недостаточно заметен у части контролов

- Категория: Accessibility
- Критичность: **Medium**
- Расположение: `templates/index.html:116`, `134-175`, `244-250`.
- Шаги: пройти интерфейс клавишей Tab.
- Ожидаемый результат: каждый интерактивный элемент имеет контрастный
  `:focus-visible`.
- Фактический результат: специальный focus есть у поиска и mode buttons,
  но у view/filter/page/logout кнопок используется слабый или браузерный
  индикатор; у select меняется только цвет рамки.
- Причина: focus-стили определены непоследовательно.
- Рекомендации: единый `:focus-visible` outline не менее 2–3 px; проверить
  порядок Tab и screen reader.

## Low

### RQA-013. Отрицательная страница молча заменяется первой

- Категория: API Validation
- Критичность: **Low**
- Шаги: `GET /api/articles?page=-100`.
- Ожидаемый результат: HTTP 400.
- Фактический результат: HTTP 200, `page=1`.
- Причина: `page = max(1, int(...))`.
- Рекомендации: отклонять значения меньше 1.

### RQA-014. Сервер не ограничивает длину поискового параметра

- Категория: API / Resource Control
- Критичность: **Low**
- Шаги: отправить `q` длиной 10 000 символов.
- Ожидаемый результат: HTTP 400/414.
- Фактический результат: HTTP 200 и полный линейный поиск по материалам.
- Причина: maxlength=500 существует только в HTML.
- Рекомендации: серверный лимит 500 и общий limit на query string.

### RQA-015. Внешняя ссылка не содержит явного rel=noopener

- Категория: Frontend Security
- Критичность: **Low**
- Расположение: `templates/index.html:683`.
- Шаги: проверить ссылку «Читать источник».
- Ожидаемый результат: `rel="noopener noreferrer"`.
- Фактический результат: есть `target="_blank"`, но нет `rel`.
- Причина: защита добавлена только в `window.open`.
- Рекомендации: добавить `rel`; не полагаться на неявное поведение браузера.

### RQA-016. Нет централизованного аудита действий пользователей

- Категория: Security Operations / Audit Trail
- Критичность: **Low**
- Шаги: войти, запустить обновление, скачать отчет.
- Ожидаемый результат: журнал user/time/IP/action/result.
- Фактический результат: имеются общие application logs, но нет надежного
  аудита входов, скачиваний и инициаторов операций.
- Причина: приложение проектировалось как локальный инструмент.
- Рекомендации: структурированный audit log с ротацией и без паролей.

### RQA-017. Изменение пользователей требует перезапуска процесса

- Категория: Administration
- Критичность: **Low**
- Расположение: `app.py:273`.
- Шаги: изменить `users.json` во время работы.
- Ожидаемый результат: безопасная перезагрузка пользователей либо admin CLI.
- Фактический результат: глобальный `USERS` загружается один раз при импорте.
- Причина: нет административного слоя.
- Рекомендации: CLI для ротации паролей и reload с lock; журнал изменений.

## Подтвержденные исправления предыдущего аудита

- стандартный Flask secret больше не используется;
- session cookie имеет HttpOnly и SameSite=Strict;
- session очищается при входе и выходе;
- CSRF защищает login, logout и refresh;
- open redirect закрыт;
- brute-force limit присутствует;
- CORS не отражает внешний Origin;
- clickjacking закрыт через `DENY` и `frame-ancestors`;
- stored DOM XSS через URL закрыт;
- Excel Formula Injection нейтрализована;
- неверные `mode`, `page` и `per_page` не вызывают HTTP 500;
- гонка записи истории закрыта lock + atomic replace;
- сервер запускается через Waitress с debug=false;
- responsive overflow отсутствует на семи разрешениях;
- зависимости не содержат известных advisories.

## UI / UX

- Горизонтальный скролл отсутствует на 320–1920 px.
- Битые изображения и неподписанные поля не обнаружены.
- Выбор режима имеет `aria-pressed`.
- Строгий и расширенный режимы не дублируют записи.
- История и Excel доступны отдельной вкладкой.
- При отключенном JavaScript показывается `noscript`.

- Основные UX-недостатки: непрозрачный долгий refresh, слабый focus у части
  кнопок и отсутствие отдельного статуса email.

подтвердить реальную доставку тестового письма и провести экспертную проверку
строгой NLP-выборки.
